Start hereWhat is Pub/Sub?

Before the tips, here's the one idea everything builds on.

Pub/Sub (short for Publish / Subscribe) is a messaging service that lets different parts of your software talk to each other without waiting on one another. One part sends out a message and immediately moves on; other parts pick it up whenever they're ready.

Analogy: Think of a radio station. The station (a publisher) broadcasts a song to a topic (a channel). Anyone with a radio tuned to that channel (a subscriber) hears it. The station doesn't know or care who's listening, and listeners don't slow the station down. That loose connection is called being decoupled.
The whole picture: Publishers drop messages into a topic. A subscription attached to that topic delivers copies to each subscriber. Everyone works at their own pace.

Four words to remember: Topic = the channel · Message = the data being sent · Publisher = sends messages · Subscriber = receives them (through a subscription).

How to send messages efficiently and reliably.

1

Batch your messages

Instead of sending each message in its own request, group several together and send them as one batch. The client libraries do this automatically by default.

Analogy: Mailing 10 letters in one envelope instead of 10 separate envelopes — cheaper and faster.
Trade-off: Batching boosts throughput and lowers cost, but a message waits a moment for the batch to fill, so it adds a little latency. Tune the batch by size, message count, or time.
PYTHON
from google.cloud import pubsub_v1

# Send a batch when it hits 100 msgs, 1 KB, OR 10 ms — whichever first
batch = pubsub_v1.types.BatchSettings(
    max_messages=100, max_bytes=1024, max_latency=0.01,
)
publisher = pubsub_v1.PublisherClient(batch_settings=batch)
2

Be careful with ordering keys

An ordering key guarantees messages are delivered in the order you sent them. Only use it when order genuinely matters (e.g. "account created" before "account updated").

Why be careful: Ordering limits how much work can run in parallel, which can slow you down. If you do use it, publish to a single region with a locational endpoint, and set up a "resume publishing" function so one failure doesn't permanently block that key.
PYTHON
# Turn ordering ON, then publish with an ordering key
opts = pubsub_v1.types.PublisherOptions(enable_message_ordering=True)
publisher = pubsub_v1.PublisherClient(publisher_options=opts)

publisher.publish(topic_path, b"order placed", ordering_key="user-123")
# after a failure, unblock that key:
publisher.resume_publish(topic_path, ordering_key="user-123")
3

Use a regional endpoint when it helps

By default Pub/Sub routes your publish to the nearest region automatically. But you can pin publishing to a specific region with a locational endpoint.

Why: This is essential for ordered messages (order is only preserved within one region) and useful for meeting data-locality requirements.
PYTHON
from google.api_core import client_options

# Pin publishing to a specific region's endpoint
opts = client_options.ClientOptions(
    api_endpoint="us-east1-pubsub.googleapis.com:443")
publisher = pubsub_v1.PublisherClient(client_options=opts)
4

Consider your bandwidth

Your publishing speed is limited by network bandwidth and how many requests you send. If bandwidth is fine but latency is high, don't flood the system with many tiny requests.

Why: Understanding your network ceiling helps you pick batch sizes and flow-control settings that actually fit your machine.
5

Provision your publishers (flow control)

Give publishers enough CPU, memory, and threads — and turn on flow control so a sudden spike of messages doesn't pile up in memory and fail.

Analogy: A bouncer at a club entrance — flow control limits how many messages are "in flight" at once so your app never gets overwhelmed.
Symptom it prevents: the dreaded Deadline exceeded errors when a publisher runs out of resources.
PYTHON
from google.cloud.pubsub_v1.types import (
    PublishFlowControl, PublisherOptions, LimitExceededBehavior)

fc = PublishFlowControl(
    message_limit=1000,
    byte_limit=10 * 1024 * 1024,            # 10 MB
    limit_exceeded_behavior=LimitExceededBehavior.BLOCK)  # wait, don't crash
publisher = pubsub_v1.PublisherClient(
    publisher_options=PublisherOptions(flow_control=fc))
6

Customize retry settings — only if needed

If a publish fails, Pub/Sub automatically retries with sensible defaults (it waits a bit longer between each attempt). You can change these, but most people never need to.

Rule of thumb: Leave defaults alone unless you're on a flaky or bandwidth-limited connection. For high volume, adjust batching and flow control first, not retry timing.
PYTHON
from google.api_core import retry

# Override the default retry only if you really need to
future = publisher.publish(
    topic_path, b"data",
    retry=retry.Retry(deadline=300))   # keep retrying up to 5 min
Bonus tip from the docs: Always check the result of a publish. The library hands back a "future" — read it asynchronously so you know whether the message actually made it, and decide what to do on failure (log, retry, or save for later).
PYTHON
future = publisher.publish(topic_path, b"data")

def on_done(f):
    try:
        print("published:", f.result())   # message ID on success
    except Exception as e:
        log.error("publish failed: %s", e)  # handle / retry / save

future.add_done_callback(on_done)        # non-blocking

How to reliably receive and process what's been sent.

1

Attach a subscription before you publish

A topic with no subscription is like shouting into an empty room — messages aren't stored and are lost forever.

Fix: Create the subscription first, or turn on topic message retention so early messages can still be replayed later.
PYTHON
subscriber = pubsub_v1.SubscriberClient()
sub_path = subscriber.subscription_path("my-app", "orders-sub")

# Attach the subscription BEFORE you publish
subscriber.create_subscription(
    request={"name": sub_path, "topic": topic_path})
2

Choose the right subscription type

Two common styles: Pull (your app asks for messages when it's ready) and Push (Pub/Sub sends messages to your web URL automatically). There are also export subscriptions that write straight to BigQuery or Cloud Storage.

Analogy: Pull = checking your mailbox when you feel like it. Push = having mail shoved through your letter slot the moment it arrives.
Tip: For most apps, use the high-level client library with a pull subscription. Choose push when your code can't run a long-lived client.
3

Process the message before acknowledging it

Acknowledging ("ack") tells Pub/Sub "I've got this, you can forget it." Do your actual work first, then ack.

Why it matters: If you ack first and then your processing crashes, the message is gone — Pub/Sub won't send it again.
PYTHON
def callback(message):
    handle(message.data)   # 1) do the real work FIRST
    message.ack()             # 2) only then acknowledge

subscriber.subscribe(sub_path, callback=callback)
4

Acknowledge messages promptly

Once a message is safely processed, ack it quickly. If you don't ack before the deadline, Pub/Sub assumes it failed and redelivers it.

Why: Slow acks cause the same messages to be delivered over and over, wasting work. Always design so your app can tolerate the occasional duplicate (Pub/Sub guarantees "at-least-once" delivery).
PYTHON
def callback(message):
    try:
        handle(message.data)
        message.ack()      # success → remove from the queue
    except Exception:
        message.nack()     # failure → ask for redelivery
5

Consider StreamingPull

StreamingPull keeps an open connection so messages flow to your subscriber continuously, instead of asking again and again.

Why: It gives the highest throughput and lowest latency for steady, high-volume workloads.
PYTHON
# subscribe() uses StreamingPull under the hood — keep it open
streaming_pull = subscriber.subscribe(sub_path, callback=callback)
with subscriber:
    try:
        streaming_pull.result()        # block & keep receiving
    except KeyboardInterrupt:
        streaming_pull.cancel()
6

Use flow control on the subscriber too

Set limits on how many messages (and how many bytes) your subscriber holds at once, so a traffic spike doesn't crush it.

Analogy: Only grabbing as many plates as you can carry, instead of letting the whole buffet land on you at once.
PYTHON
# Hold at most 100 messages / 10 MB in flight at a time
flow = pubsub_v1.types.FlowControl(
    max_messages=100, max_bytes=10 * 1024 * 1024)
subscriber.subscribe(sub_path, callback=callback, flow_control=flow)

1

Disable client-side batching for lowest latency

Batching saves cost but adds delay. If your priority is getting each message out as fast as possible, turn batching off.

Why: It's the same trade-off in reverse — speed over efficiency. Pick based on what your app needs.
PYTHON
# max_messages=1 → every message is sent immediately (no waiting)
batch = pubsub_v1.types.BatchSettings(max_messages=1)
publisher = pubsub_v1.PublisherClient(batch_settings=batch)
2

Consider regions

Publish and process close to where your users and data live to cut down the round-trip time.

3

Increase reliability with regional isolation

Spreading work across regions means a problem in one region doesn't take your whole system down.

Analogy: Don't keep all your eggs in one basket — if one drops, the others survive.
4

Watch the delivery latency health score

Pub/Sub gives you a health score that tells you whether messages are flowing smoothly.

Why: It's an early-warning light — check it before small slowdowns become outages.
5

Use the Pub/Sub metrics dashboards

Built-in dashboards in Cloud Monitoring show throughput, backlog, and errors at a glance.

Why: You can't fix what you can't see. Monitoring is how you spot a growing backlog of unprocessed messages.
6

Set up quota alerts

Get notified before you hit Pub/Sub usage limits.

Why: Hitting a quota cap unexpectedly can stall your pipeline. An alert buys you time to react.

Using the SDK wellClient Library Tips

Small habits in your code that make Pub/Sub run smoothly. (Libraries exist for Python, Java, Go, Node, C++, and more.)

Let the library do the work: the official client library sits between your code and Pub/Sub and handles the hard parts for you — so reusing it and keeping it updated pays off.

1

Pick the right client library language

Use the official high-level library for your language rather than calling the raw API yourself.

Why: The libraries handle batching, flow control, retries, and acks for you — so you get the best practices "for free."
BASH
pip install google-cloud-pubsub
PYTHON
from google.cloud import pubsub_v1

publisher  = pubsub_v1.PublisherClient()   # high-level clients
subscriber = pubsub_v1.SubscriberClient()
2

Reuse your publisher client

Create one publisher object and keep using it — don't make a new one for every message.

Analogy: Keep one phone line open instead of dialing a fresh call for every sentence.
PYTHON
publisher = pubsub_v1.PublisherClient()   # create ONCE

for order in orders:
    publisher.publish(topic_path, order)   # reuse for every message
3

Use flow control

Same idea as before — let the library cap how much is in flight so your app stays stable under load.

4

Always update to the latest version

Keep your client library current.

Why: Newer versions bring performance improvements, bug fixes, and security patches.
BASH
pip install --upgrade google-cloud-pubsub
5

Use async properly (especially in Node.js)

Lean on the language's asynchronous features so your code doesn't block while waiting on messages.

Why: Async handling means one slow message doesn't freeze everything else.
PYTHON
# Fire many publishes without blocking, then wait at the end
futures = [publisher.publish(topic_path, m) for m in messages]

for f in futures:
    f.result()   # they were sent concurrently, not one-by-one
6

Understand lease management

When your subscriber receives a message, it holds a "lease" — a temporary deadline to finish processing. The library can extend this automatically for messages that take a while.

Why: Proper lease handling stops long-running tasks from being redelivered just because they took a few extra seconds.
PYTHON
# Let the library auto-extend the lease for up to 10 minutes
flow = pubsub_v1.types.FlowControl(max_lease_duration=600)
subscriber.subscribe(sub_path, callback=callback, flow_control=flow)

1

Push subscriptions

Have Pub/Sub deliver messages directly to your web endpoint (a URL) as they arrive — great with serverless functions.

PYTHON
push = pubsub_v1.types.PushConfig(
    push_endpoint="https://myapp.com/pubsub/push")
subscriber.create_subscription(request={
    "name": sub_path, "topic": topic_path, "push_config": push})
2

Seek

"Rewind" or "fast-forward" your subscription to a point in time — replay old messages or skip a backlog.

Analogy: The seek bar on a video player, but for your message stream.
PYTHON
from datetime import datetime, timedelta, timezone

# Rewind this subscription to one hour ago
one_hour_ago = datetime.now(timezone.utc) - timedelta(hours=1)
subscriber.seek(request={"subscription": sub_path, "time": one_hour_ago})
3

Message retention in the topic

Store messages at the topic level so any subscription — even one created later — can replay them.

PYTHON
from google.protobuf.duration_pb2 import Duration

publisher.create_topic(request={
    "name": topic_path,
    "message_retention_duration": Duration(seconds=7*24*3600)})  # 7 days
4

Message retention in the subscription

Keep already-acknowledged messages around for a set time, in case you need to replay them with seek.

PYTHON
subscriber.create_subscription(request={
    "name": sub_path, "topic": topic_path,
    "retain_acked_messages": True,
    "message_retention_duration": Duration(seconds=3*24*3600)})
5

Dead-letter topic

If a message keeps failing after several tries, send it to a separate "dead-letter" topic instead of retrying forever.

Analogy: The undeliverable-mail bin at the post office — problem messages get set aside so they don't block everything else.
Why: You can inspect failures later without losing data or jamming your pipeline.
PYTHON
dlq = pubsub_v1.types.DeadLetterPolicy(
    dead_letter_topic=dlq_topic_path,
    max_delivery_attempts=5)        # give up after 5 tries
subscriber.create_subscription(request={
    "name": sub_path, "topic": topic_path, "dead_letter_policy": dlq})
6

Configure by region

Use a message storage policy to control which regions your message data is stored in.

Why: Essential for compliance — e.g. a European company keeping all data inside the EU.
PYTHON
# Only ever store this topic's data in EU regions
storage = pubsub_v1.types.MessageStoragePolicy(
    allowed_persistence_regions=["europe-west1", "europe-west3"])
publisher.create_topic(request={
    "name": topic_path, "message_storage_policy": storage})

Habits that keep a growing system clean, secure, and testable.

1

Use unique, descriptive naming

Give topics and subscriptions clear, consistent names.

Why: Future-you (and your teammates) will instantly know what orders-prod does versus x1-test.
2

Avoid the high fan-out pattern

Be cautious about having one topic feed a huge number of subscriptions at once.

Why: Very high fan-out can spike resource use and cost. Design deliberately.
3

Use granular subscriber credentials (IAM)

Give each subscriber only the access it actually needs — not blanket permissions.

Analogy: A hotel key card that opens only your room, not every door in the building.
BASH
# Grant ONE service account just the Subscriber role on ONE topic
gcloud pubsub topics add-iam-policy-binding payments \
  --member="serviceAccount:payments-sa@my-app.iam.gserviceaccount.com" \
  --role="roles/pubsub.subscriber"
4

Aim for consistent throughput

Steady, predictable message rates are easier to scale for than wild spikes.

5

Test locally with the Pub/Sub emulator

The emulator lets you run Pub/Sub on your own machine — no cloud project or cost needed.

Why: Develop and test safely before touching real, billable resources.
BASH
gcloud beta emulators pubsub start --project=my-app
export PUBSUB_EMULATOR_HOST=localhost:8085
# your normal Python code now talks to the LOCAL emulator — no cost
6

Add labels

Tag your resources with labels (like team:payments or env:prod).

Why: Labels make usage reporting and cost tracking far easier.
PYTHON
publisher.create_topic(request={
    "name": topic_path,
    "labels": {"team": "payments", "env": "prod"}})
7

Consume messages by region

Process messages close to where they're stored to reduce latency and cross-region cost.

8

Set a sensible retention duration

Pick how long messages are kept based on how long you might realistically need to replay them.

Why: Too short risks losing data you need; too long can raise storage costs.

ReferenceGlossary & Quick Checklist

Topic
The named channel publishers send messages to.
Message
The piece of data being sent (plus optional attributes).
Publisher
Code that sends messages to a topic.
Subscriber
Code that receives messages through a subscription.
Subscription
The link between a topic and a subscriber; it holds messages until they're acknowledged.
Acknowledge (ack)
Telling Pub/Sub a message was successfully processed so it won't be redelivered.
Batching
Grouping multiple messages into one request for efficiency.
Flow control
Limiting how many messages are "in flight" so apps don't get overwhelmed.
Pull vs. Push
Pull = subscriber requests messages; Push = Pub/Sub sends them to a URL.
StreamingPull
A continuous open connection for high-throughput pulling.
Lease
The temporary deadline a subscriber has to process a message.
Seek
Rewinding or skipping ahead in a subscription's message timeline.
Dead-letter topic
A holding place for messages that repeatedly fail processing.
At-least-once delivery
Pub/Sub's promise that every message arrives — possibly more than once, so handle duplicates.

The one-page checklist

  • Attach a subscription (or enable retention) before publishing.
  • Let the official client library handle batching, retries & acks.
  • Process messages before you acknowledge them.
  • Turn on flow control on both publishers and subscribers.
  • Only use ordering keys when order truly matters.
  • Choose batching (efficiency) vs. no batching (low latency) on purpose.
  • Reuse one publisher client; keep the library up to date.
  • Add a dead-letter topic for messages that keep failing.
  • Monitor dashboards, the health score, and set quota alerts.
  • Name resources clearly, add labels, and test with the emulator.
  • Design so your app tolerates occasional duplicate messages.
